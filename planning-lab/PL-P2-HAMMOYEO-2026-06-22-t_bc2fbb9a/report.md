## 1. 한 줄 결론

`함모여`는 단순 일정 투표 서비스가 아니라 **방장이 단체 모임을 실제로 확정하도록 돕는 경량 모임 합의 엔진**으로 포지셔닝하는 것이 타당합니다. 다만 MVP는 날짜/시간 확정과 차선안 추천을 가장 깊게 해결하고, 장소·활동·식사는 “정교한 추천”이 아니라 결정 보조 수준으로 제한하는 편이 안전합니다.

## 2. 제품 정체성 제안

권고 정체성: **모임 확정 자동화 / 합의 엔진**

- 일정 투표 서비스: 가능 시간을 모으고 표를 보여주는 도구입니다.
- 함모여가 지향해야 할 정체성: 가능 시간, 선호 강도, 조정 가능성, 최소 장소/활동 조건을 합쳐 **방장이 바로 공유할 수 있는 약속안 1~3개**를 제안하는 도구입니다.
- 핵심 문장: “다들 언제 돼?”에서 끝나는 앱이 아니라, “그럼 이 안으로 보자”까지 도달하게 하는 앱입니다.

제품 약속은 크게 잡되, MVP 구현 깊이는 좁혀야 합니다. “모임 확정”이라는 차별성은 유지하되, 초기에는 **날짜/시간 합의 → 현실적 차선안 → 공유 가능한 최종안**에 집중하는 것이 좋습니다.

## 3. 핵심 문제 / 타깃 정리

### Primary user

단체 모임을 만들고 마지막 결정을 떠안는 **방장 / 주최자**입니다.

### Secondary user

초대 링크를 받아 가능 시간과 간단한 선호를 빠르게 입력해야 하는 **참여자**입니다.

### 핵심 상황

친구 모임, 동호회, 소규모 팀, 가족 모임 등에서 단체 채팅방으로 날짜, 시간, 지역, 음식/활동을 정하려 하지만 대화가 길어지고 최종 확정이 미뤄지는 상황입니다.

### 핵심 고통

- 방장이 응답 수집, 미응답자 재촉, 결과 해석, 절충안 도출, 최종 공지를 모두 떠안습니다.
- 전원 가능 시간이 없으면 논의가 중단되거나 다시 처음부터 시작됩니다.
- 날짜를 정해도 장소/식사/활동 논의가 새로 시작됩니다.
- 참여자는 복잡한 입력이나 반복 질문 때문에 응답을 미룹니다.

## 4. MVP 범위 권고

### Must include

1. 약속방 생성 및 초대 링크 공유
2. 후보 날짜/시간 자동 생성 또는 방장의 빠른 설정
3. 참여자의 10~30초 내 가용성 입력
4. 단순 가능/불가능이 아닌 선호 강도 입력
   - 딱 좋아요
   - 가능해요
   - 조정 가능해요
   - 절대 안 돼요
5. 전원 가능 시간이 없을 때도 멈추지 않는 차선안 추천
6. 추천안의 이유 표시
   - 참석 가능 인원
   - 조정 필요 인원
   - 강한 불가 인원
   - 선호 강도
7. 방장이 바로 확정하고 공유할 수 있는 최종안 문구
8. 장소·활동·식사는 상세 추천이 아니라 최소 조건 수집 / 카테고리 수준 제안

### Should defer

1. 정확한 식당·카페·장소 추천
2. 예약, 결제, 지도 경로 최적화
3. 교통 API 기반 정밀 이동시간 계산
4. 캘린더 양방향 연동
5. 자연어/AI 입력을 핵심 입력 방식으로 삼는 것
6. 취향 기반 개인화 추천 엔진
7. 정산, 출석, 복잡한 권한 관리

### Chair 판단

장소·활동·식사를 완전히 빼면 기존 일정 조율 도구와의 차별성이 약해질 수 있습니다. 반대로 깊게 넣으면 MVP가 과도해집니다. 따라서 P2 결론은 다음 절충안입니다.

- 핵심 깊이: 날짜/시간 확정과 차선안 추천
- 보조 폭: 장소·활동·식사 조건은 얕게 포함
- AI/자연어: 초기 핵심이 아니라 보조 입력 옵션

## 5. 주요 사용자 흐름

1. 방장이 모임 이름, 대략적 기간, 선호 시간대, 예상 참여자 또는 인원을 입력합니다.
2. 함모여가 후보 날짜/시간을 자동 생성하고, 방장이 필요 시 수정합니다.
3. 방장이 초대 링크를 단체 채팅방에 공유합니다.
4. 참여자는 로그인 부담 없이 각 시간대에 대해 선호 강도와 가능 여부를 입력합니다.
5. 필요 시 참여자는 출발 지역, 피하고 싶은 음식/활동, 간단한 조건을 추가합니다.
6. 함모여가 응답률, 선호 강도, 조정 가능성, 불가능 인원을 반영해 추천 약속안 1~3개를 랭킹으로 보여줍니다.
7. 전원 가능 시간이 없으면 가장 손실이 적은 차선안과 조정 대상자를 함께 보여줍니다.
8. 방장이 추천안 중 하나를 확정하고, 공유 가능한 최종 문구를 단체 채팅방에 붙여 넣습니다.
9. 참여자는 확정안을 확인합니다.

## 6. 차별점

1. 가능 시간 표를 보여주는 데서 끝나지 않고 **확정 가능한 추천안**을 제시합니다.
2. 전원 가능 시간이 없어도 중단하지 않고 현실적인 차선안을 냅니다.
3. 방장의 해석 부담을 줄이는 추천 이유와 순위를 제공합니다.
4. 단순 가능/불가능이 아니라 선호 강도와 조정 가능성을 반영합니다.
5. 날짜 확정 이후 장소·활동·식사 논의가 다시 시작되는 문제를 최소 조건 수집으로 완화합니다.
6. 참여자 입력을 10~30초 안에 끝내는 낮은 마찰 구조를 우선합니다.

## 10. Council 통합 기록

### Round 1

4개 Council 의견을 동일 입력 패킷과 동일 JSON 스키마로 수집했습니다.

### Round 2

의견은 `Opinion A/B/C/D`로 익명화했습니다. 모델 매핑은 별도 audit 파일에만 저장했습니다.

공통 합의:

- 제품 정체성은 일정 투표보다 모임 확정 자동화/합의 엔진이 적절합니다.
- Primary user는 방장입니다.
- MVP 핵심은 날짜/시간 확정과 차선안 추천입니다.
- 장소·활동·식사는 초기 차별성을 보여주는 보조 수준으로 제한해야 합니다.
- 자연어/AI 입력은 초기 핵심이 아니라 보조 기능으로 두는 것이 안전합니다.
- 다음 단계는 P3가 적절합니다.

주요 뉘앙스 차이:

1. 장소·활동·식사를 MVP에 어느 정도 포함할지
   - 일부 의견은 카테고리 수준 보조 포함을 권고했습니다.
   - 일부 의견은 상세 장소/활동 추천은 명확히 이연해야 한다고 보았습니다.
   - Chair 결론: “최소 조건 수집 + 카테고리 제안”까지 포함하고, 실제 업체/정밀 이동 최적화는 이연합니다.
2. 자연어/AI 입력의 위치
   - 모든 의견이 초기 핵심 입력으로 두는 것은 위험하다고 봤습니다.
   - Chair 결론: 선택형 입력이 기본, 자연어는 보조/확인 흐름으로 둡니다.
3. 제품 성장성
   - 한 의견은 독립 서비스로서 사용 빈도와 지불 의사가 낮을 수 있다는 반대 근거를 강하게 제기했습니다.
   - Chair 결론: P3에서 반복 사용 동기와 전환 동기를 반드시 검증해야 합니다.

### Round 3 제한 반박 여부

명시적 정면 충돌은 크지 않아 별도 Round 3 상호 반박은 실행하지 않았습니다. 대신 위 3개 뉘앙스 차이를 Chair가 쟁점으로 묶어 P2 결론에 반영했습니다.

## 11. 주요 근거

- 제공된 초안은 이미 “가능 시간 표”보다 “현실적 약속안 자동 추천”을 핵심으로 두고 있습니다.
- 기존 일정 조율 서비스의 약점은 가능 시간 수집 이후 최종 판단과 장소/활동 논의가 사용자에게 남는다는 점입니다.
- 방장은 응답 수집, 결과 해석, 차선안 도출, 최종 공지 부담을 가장 크게 집니다.
- “전원 가능 시간이 없을 때도 멈추지 않는 차선안 추천”이 기존 투표 도구와의 핵심 차별점이 될 수 있습니다.
- 다만 장소/식사/활동 추천을 깊게 구현하면 MVP 범위가 과도해져 핵심 가설 검증이 흐려질 가능성이 큽니다.

## 12. 핵심 반대 의견

가장 강한 반대 의견은 다음입니다.

> 단체 약속 조율은 불편하지만 사용 빈도와 지불 의사가 낮아 독립 서비스로 성장하기 어렵고, 카카오톡·캘린더·기존 투표 도구 안에서 해결될 가능성이 높습니다. 또한 MVP가 날짜/시간에 집중하면 기존 일정 투표와 차이가 약하고, 장소·활동·식사까지 깊게 넣으면 복잡도가 커질 수 있습니다.

이 반대 의견 때문에 P4 직행은 권고하지 않습니다. P3에서 실제 전환 동기와 추천안 채택률을 확인해야 합니다.

## 13. 남은 불확실성

- 방장이 실제로 추천안을 신뢰하고 그대로 공유하는가?
- 참여자 응답률이 별도 링크 환경에서도 충분히 나오는가?
- 장소·활동·식사 보조 기능의 최소 깊이는 어디인가?
- 자연어 입력이 체감 편의성을 높이는가, 아니면 오류와 확인 비용을 늘리는가?
- 반복 사용과 수익화 가능성이 있는 세그먼트가 어디인가?

## 14. Chair 결론

`함모여`는 P2 기준에서 **제품화 검토를 계속할 가치가 있습니다.** 다만 지금 단계의 올바른 결론은 “최종 기획 확정”이 아니라 “P3에서 좁은 핵심 가설을 검증할 만하다”입니다.

P3 권고 범위:

- 타깃 세그먼트 1~2개 선정
- 방장 인터뷰 또는 시나리오 테스트
- 참여자 10~30초 입력 플로우 검증
- 추천안 1~3개가 실제 선택되는지 테스트
- 장소·활동·식사 보조 깊이 검증
- 기존 도구 대비 전환 이유 검증
